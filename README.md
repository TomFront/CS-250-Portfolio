# CS-250-Portfolio

- How do I interpret user needs and implement them into a program? How does creating "user stories" help with this?
Interpreting user needs starts with listening carefully to what the client wants and translating those wants into actionable development tasks. User stories made this process much more concrete — by framing requirements as "As a user, I want to... so that...", I was able to focus on the actual value being delivered rather than just technical features. They helped keep the team aligned with the client's goals throughout each sprint.

- How do I approach developing programs? What Agile processes do I hope to incorporate into my future development work?
I approach development iteratively, breaking work into manageable pieces and revisiting requirements as the project evolves. Through CS-250, I came to appreciate how Agile practices like sprint planning, daily standups, and retrospectives keep a team focused and adaptable. Going forward, I want to continue using these structures, especially retrospectives, since regularly reflecting on what's working and what isn't leads to meaningful improvement over time.

- What does it mean to be a good team member in software development?
Being a good team member means communicating openly, following through on commitments, and supporting others when they're stuck. In a Scrum environment especially, each role — Developer, Tester, Product Owner, Scrum Master — depends on the others. I learned that transparency and accountability aren't just professional courtesies; they're what allow a team to actually deliver working software on time.
